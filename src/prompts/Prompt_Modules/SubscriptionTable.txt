Reconciliation tables: Any time you have a set of codes that starts with a code with a label like "Beginning" and ends with a label like "Ending", you are dealing with a "reconciliation table" and there are some standardized things to look out for. Go through each of the items below and look through every input driver in each reconciliation table to make sure these mistakes are not occurring:
a.	The beginning row should always reference the ending row below it. In the example below, <OFFSETCOLUMN-S; sumif="Offsetyear"; driver1="R2"; format="Volume"; indent="2"; bold="false"; topborder="false"; italic="true"; row1 = "V3|Beginning|||||F|F|F|F|F|F|";> references input driver R2, which is the output driver of the row labeled ‚ÄúEnding‚Äù. The encoder step commonly messes this up. Always verify that each ending row is linked to the driver of the beginning row. 
b.	The ending row should always be a SUM2-S code that references the beginning row of the table and the row directly above it. In the example, the row labeled ‚ÄúEnding‚Äù references the beginning row, which has an outputdriver of ‚ÄúV3‚Äù and the row labeled ‚ÄúLost‚Äù, which has an output driver of R1 and is the row directly above the Ending row. Thus, the sum formula will sum all of the items from V3 to R1. <SUM2-S; driver1="V3"; driver2="R1"; bold="true"; topborder="true"; format="Volume"; indent="1"; italic="true"; sumif="Yearend"; row1 = "R2|Ending|0||||F|F|F|F|F|F|";>. The encoder step commonly messes this up. Verify that driver1 and driver2 are at either ends of the range that needs to be summed. 
c.	There must not be any subtotals in between beginning and ending. 
d.	The sumif parameter for beginning should always be ‚Äúoffsetyear‚Äù
e.	The sumif parameter for ending should always be ‚Äúyearend‚Äù
f.	The sumif parameter for the items in between should be ‚Äúyear‚Äù
g.  Upgrades/downgrades between two tables (tiers of subscribers, etc.) should have negative=‚Äùtrue‚Äù in both rows! First MULT calc multiplies by the percentage upgrade/downgrade and must be turned negative in order to account for subs leaving the table. In the next table, the downgrade/upgrade row will use DIRECT-S to directly reference the MULT calc in the other table. Because that calc is already negative, you must turn it positive by setting negative=‚Äùtrue‚Äù and thus creating a double negative which results in a positive. 
h. the label for % upgrade/% downgrade must include time period. If the assumption is that X% per year will downgrade, use SPREAD-E and no time period in the label. If the assumption is that X% per month will downgrade/upgrade, use CONST-E and "% Upgrade/Downgrade per Month" as the label.
i. **BEGIN-S:** Takes ending value of last period. Use as beginning row of reconciliation tables. Driver1 should be END-S row1
j. **END-S:** Sums reconciliation table. Set driver1 to BEGIN-S row1. These codes should ALWAYS reference each other
k. **üö® CRITICAL END-S LABELING RULE:** DO NOT include any wording about "Ending" in the label for END-S. This should always be the pure name of the item. Ex: if the table is for # of subscribers, the END-S label should be "# of Subscribers", NOT "Ending # of Subscribers" or "# of Subscribers Ending"
l. **ACTUAL CODE INTEGRATION:** When given information about beginning # of Subscribers, add fincode "V: volume" to (F) of END-S and use ACTUAL code to add beginning # of subscribers. Structure should be:
